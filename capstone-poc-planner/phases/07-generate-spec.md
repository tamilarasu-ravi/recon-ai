# Phase 7 — Generate the POC spec

## Purpose

Produce the final markdown POC specification. This is the artifact the student walks away with — it captures everything from Phases 0–6 in a form they can hand back to Claude (or any tool) when they start building.

## Preconditions

- Phases 0–6 complete with all output contracts satisfied.
- Student approved: idea (Phase 1), sources (Phase 2), tech stack (Phase 5), eval plan (Phase 6).

## Process

Generate a single markdown file. Use **exactly the structure below**, filling each section from the conversation so far. Save the file to the workspace as `<project-slug>-poc-spec.md`.

### The spec template

```markdown
# [Project name] — POC Specification

> Generated [date]. Hand this spec to Claude (or another tool) to begin building. It contains enough context to start cleanly without re-explaining the project.

## 1. One-paragraph summary

[2–4 sentences: what it does, who it's for, why it's interesting. From Phase 0 + 1.]

## 2. The problem and the user

- **Specific user**: [concrete user segment from Phase 3]
- **Problem**: [what they struggle with today]
- **Current alternative**: [what they do without this]
- **Wedge**: [why this approach is different / better — from Phase 1 question 2]

## 3. Why now

[What recent capability makes this possible. One paragraph. From Phase 1 question 3.]

## 4. Competitive landscape

| Existing solution | Approach | How this project differs |
|---|---|---|
| [from Phase 2 sources] | ... | ... |
| ... | ... | ... |

## 5. Capability-trajectory assumptions

- **Assumes**: [list model capabilities the project depends on, from Phase 3]
- **Survives improvement when**: [how it gets better as models improve]
- **At risk if**: [what model-capability change would obsolete it]

## 6. POC scope

- **In scope for POC**: [bullet list — keep tight]
- **Explicitly out of scope**: [bullet list — protects against scope creep]
- **Smallest hypothesis to prove**: [one sentence — the single thing the POC must demonstrate]

## 7. Tech stack

- **Model(s)**: [provider + model IDs, from Phase 5]
- **Agent framework**: [or "raw API"]
- **Retrieval stack**: [if applicable — vector DB, embedding model, reranker, hybrid/dense]

- **Orchestration / hosting**: [where it runs]
- **Frontend**: [if any]

## 8. Architecture sketch

[Plain-text or ASCII flow of the main path. User input → ... → output. Keep it under 12 lines. If hybrid (e.g., RAG + agentic), show both paths.]

## 9. Eval plan

- Judge model: [...]

## 10. Risks and mitigations

| Risk | Likelihood (L/M/H) | Mitigation |

The smallest thing to build first — what the student should do in their first week of building to prove the core hypothesis. Five or fewer concrete tasks.

2. ...

## 13. Sources used in planning

[Numbered list of every primary source from Phase 2, with URLs and one line each on what each contributed. Include the "couldn't find primary on" list if any.]

2. ...

[Anything the student should resolve before building, or research deeper. From any phase where uncertainty was flagged.]

---

*Generated by the capstone-poc-planner skill. Hand this spec to Claude with "Build the POC described in this spec" to start a clean build session.*
```

### After generating

1. Save the file to the workspace.
2. Tell the student where it is and what they can do with it:
   - Hand it back to Claude (or Claude Code) to start the build
   - Use section 12 (Week-1 plan) as their immediate to-do list
   - Use section 9 (Eval plan) to build the eval harness *in week 1*, not at the end
3. Remind them: the spec is reusable. If their build session goes off the rails, returning to the spec will get them back on track.

### Skipped or weak sections

If any phase was skipped or rushed (e.g., the student insisted on jumping ahead), add a note at the bottom of the spec:

```
## Note on planning completeness

The following sections were filled in without full Phase-by-Phase planning:
- [section] — [why]

The student should treat these sections with extra caution and plan to revisit them before committing to the build.
```

## Output contract

Before this phase is done:

- [ ] Spec file generated at the workspace with all 14 sections
- [ ] Every section has content (not "TBD") — if a section genuinely doesn't apply (e.g., no LLM-judge for a deterministic project), state so explicitly rather than leaving blank
- [ ] Sources list is complete and matches what was approved in Phase 2
- [ ] Student has been told where the file is and how to use it next

## Project-type notes

- **RAG-heavy**: section 8 (architecture sketch) should show the retrieval path explicitly — chunking → embedding → retrieval → reranking → generation. Section 9 should include retrieval recall as a metric.

- **Agentic**: section 8 should show the tool-use loop. Section 9 must include trajectory / tool-call evaluation.

- **Fine-tuning**: section 7 should include training framework and dataset format. Section 11 should call out training compute separately from inference compute. Section 9 should compare against the base model as a baseline.

- **Multimodal**: section 7 should specify which modalities each model handles. Section 9 should include modality-specific failure modes.

- **Inference/systems**: section 9 must include a baseline (e.g., "vs default vLLM config"). Section 11 should include target hardware specs.
